1. Core Design Principles

The platform splits core compute execution patterns into three critical operational boundaries to achieve deterministic vehicle response profiles under harsh environmental conditions.

Boundary 01

Isolation of Safety Paths

Separates AI perception inference pipelines from actuation watchdogs and functional emergency signals.

Boundary 02

No Zero-Copy Latency Penalties

CPU and GPU in-vehicle memory spaces share unified hardware registers to eliminate standard frame transport overheads.

Boundary 03

Asymmetric Run Optimization

Saves massive hardware operations outlays by targeting AWS Graviton3 and GCP spot instances on validation stacks.

Refer to "Svashasan OS Deployment Compute Architecture.pdf" for structural deep-dives and detailed network layout drawings relating to in-vehicle CAN configurations.

2. Tier 1: In-Vehicle Real-Time Edge

This represents the absolute focus of Svashasan OS Core. Real-time perception and immediate driving trajectory decisions execute directly inside the vehicle payload cabinet under strict power constraints.

Specification ParameterTarget Architecture DetailsStatus
Edge Compute Core NVIDIA Jetson AGX Orin (ARM Cortex-A78AE + Ampere GPU)
Memory Management Unified Memory Pool (64GB LPDDR5) - zero-copy GPU memory passing
System Software Base Custom Yocto BSP Layers mapped directly to real-time kernel modules
Autonomous Middleware ROS2 Humble nodes operating on high-speed internal IPC systems
Model Optimization TensorRT compilation utilizing mixed-precision FP16 and INT8
Avoid x86 Architectures inside the Vehicle

x86 CPUs draw upwards of 35W before computing single neural inference frames. They require heavy mechanical active-cooling setups which are highly prone to physical failure in shaking vehicular cabinets. High-power draw and high-vulnerability components should be isolated strictly to Tier 2 environments.

3. Tier 2: Cloud Training & Validation

Designed to run massive data pipelines, train primary neural models, run high-fidelity simulations (CARLA), and securely serve internal API structures.

Simulation & Heavy Training

Platform: GCP / x86 Architecture + GPUs

CARLA Simulation: Run inside GCP n2-standard-32 environments to process complex, high-frequency single-thread physics.

Primary Deep Training: Leverages GCP GPU clusters with A100/H100 cards to perform heavy multi-sensor model training loops.

Inference & Run Optimization

Platform: AWS / Graviton3 ARM Architecture

Inference Runtimes: Microservices run within c7g.2xlarge environments to achieve immediate 40%+ pricing benefits over legacy x86 nodes.

Heavy ETL Processing: Data manipulation pipelines utilize m7g.xlarge clusters to lower overall data operations run costs.

Spot & Preemptible Integration Savings: 60% – 80% Reduction on Validation Clusters

4. Tier 3: Real-Time Safety MCU Island

Designed to execute absolute worst-case safe vehicle states. This tier is physically and electrically isolated from the main edge SoC systems to prevent complex OS kernel locks from inhibiting functional stopping commands.

Primary Safety Responsibilities

  • Continuous hardware watchdog monitoring of the primary NVIDIA Jetson Orin SoC.
  • Direct management of CAN-FD network packets communicating vehicle velocity commands to braking actuators.
  • Autonomous hardware emergency signal override in <1ms if the primary AI engine fails.
Target MCU Core Topologies

ARM Cortex-R52 (Current Target)

Proven, automotive-grade silicons featuring full ASIL-D certification. Extremely mature software toolchain support.

RISC-V Automotive Core (Roadmap Path)

EU Horizon Europe CHIPS funded implementations. Tracked for migration when safety-certified chips mature.

6. Heterogeneous Rationale

AV deployment groups occasionally design architectures targeting a single ISA for simplicity. The table below details why a single-ISA approach fails production standards.

Alternative StrategyOperational RoadblockDirect Impact
Pure x86 Everywhere No embedded hardware NPU, unsustainable power draw at the edge. Fails vehicle thermal limits.
Pure ARM Everywhere Lacks PCIe GPU bus bandwidth in the cloud for training massive vision models. Severely throttles training pipelines.
No Dedicated Safety Island An OS freeze on the primary SoC leaves no backup controller for vehicle actuation. Violates ISO 26262 ASIL-D.

7. Immediate Actions

These priority actions are designed to move the platform from TRL 6 (Simulation-Verified) to TRL 7 (In-Vehicle Live Test Bed) using cost-efficient development models.

01

Procure NVIDIA Jetson AGX Orin Developer Hardware

Acquire developer hardware using the NVIDIA Inception program discount. This setup forms the primary physical target for the custom OS kernel layers.

02

Port Custom Yocto Linux Layers to ARM Target

Configure base Yocto Project recipes to align drivers specifically to the Jetson ARM Cortex-A78AE safety-extended cores.

03

Move Simulation Runs to GCP Spot Instances

Configure CARLA simulation jobs to execute on GCP Preemptible VMs, capturing immediate 60%–80% operational cost savings.

04

Compress Perception Models via TensorRT

Port existing PyTorch model layers into quantized TensorRT runtimes. Aim for 3x–5x performance gains over unquantized GPU execution.

8. Technical Glossary

TermDefinition & AV Application
ASIL-D Automotive Safety Integrity Level D. Represents the highest safety hazard classification assigned within the ISO 26262 functional safety framework.
TOPS Tera Operations Per Second. Standard mathematical unit tracking raw computing hardware execution performance limits for AI models.
BSP Board Support Package. The customized collection of device driver configurations, boot loaders, and microcode configurations required to run specific hardware architectures.
CAN-FD Controller Area Network with Flexible Data-Rate. High-performance automotive bus protocol expanding data payload capacities beyond legacy CAN standards.

System Integration Verdict

The heterogeneous compute architecture plan defined in svashasan-compute-architecture.html provides a structurally compliant path for scaling Svashasan OS to vehicle-ready status. By cleanly separating heavy training (GCP/x86), edge inference (AGX Orin), and critical control safety loops (Cortex-R52), the design satisfies both the strict real-time latencies of AV systems and the commercial cost constraints of a scaling startup.
